  VERTEX  = The consensus engine (the radio technology inside)
            Uses Hashgraph DAG to reach agreement in <100ms.

            This is what your drones actually use to talk.
            Speaks standard MQTT — familiar publish/subscribe.
            Your nova/tasks, nova/worldstate, nova/heartbeat

  M1 focuses on drone MQTT clients and topic design.

HOW FOXMQ WORKS FOR M1 (simple explanation):
  - FoxMQ is an MQTT broker, but decentralized.
  - In a normal MQTT setup: all devices connect to ONE broker
    server (like RabbitMQ or Mosquitto). If that server dies,
    everyone loses connection.
  - In FoxMQ: the broker itself is distributed across all nodes.
    Every drone IS a broker node. No single server can kill it.
  - M1's job: spin up FoxMQ on both Linux machines and connect
    drone MQTT clients to the local FoxMQ mesh.

KEY CHANGE FROM v1:
  M2's auction and CRDT now use FoxMQ topics for all messaging.
  All publish/subscribe calls go through M1's nova_mesh.py.
  Fair Ordering guarantee: FoxMQ ensures ALL drones receive
  auction messages in the EXACT same order — critical for
  preventing two drones from winning the same task bid.

HOW FOXMQ HELPS M2 (important to understand):
  - Normal MQTT: message order depends on when they hit server.
    Network jitter can change order. Two drones might both
    think they won the same bid.
  - FoxMQ consensus-backed queue: every subscriber receives
    messages in mathematically identical order. This makes
    M2's auction engine reliable without extra locking code.
